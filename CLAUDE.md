# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This is **Ressam**, a tabletop RPG (TTRPG) rulebook written in Markdown. Most of the repository is content/rules-authoring - edits to game design text, tables, and formulas, not software. However, the weapons/armor tables, spell lists, feat lists, and the character progression table are **generated from structured YAML data** via a small Python + Jinja2 pipeline (see "Data-driven content pipeline" below) - those specific files are build output, not hand-authored prose. A separate, read-only **balance analysis tool** (see "Balance analysis tooling" below) reuses that same structured data plus a small formula/archetype layer to compute derived-stat progression curves (attack bonus, Evasion, AR, HP, spell modifier, ...) across levels - it does not generate or modify anything under `core/`.

## Repository structure

- `core/core_rules.md` - foundational mechanics: dice, attributes, skills, checks vs. Ward, HP/dying, carrying slots, resting, exhaustion, stealth, light/vision. Read this first; nearly every other chapter assumes these definitions.
- `core/character/` - building and advancing a character: `character_creation.md` (step-by-step guide), `races/` (one hand-authored file per race, indexed by `races_overview.md` - see below), `careers.md`, `languages.md`, `progression_&_rewards.md` (**generated**, see below - the per-level advancement table renders from data, and the rest of the file - Rewards & Treasure, Itemization - is static prose baked into the same template, the same pattern `weapons.md.j2` uses for its glossary/rules sections).
  - `core/character/races/` - `races_overview.md` (hand-authored: General Rules Notes, Level Adjustment, and an index of races grouped by Artefolk/Cursed/Mystik/Night-Terror) plus one hand-authored file per race (`aelves.md`, `dwergaz.md`, `humans.md`, `orkhs.md`, `alsahli.md`, `feliids.md`, `skeggs.md`, `tembels.md`, `khoridae.md`, `dzinari.md`, `golems.md`, `tapio.md`, `daemons.md`, `varulf.md`, `strygs.md`). This split (one page per race, mirroring `core/magic/arcane/` and `core/magic/divine/`) exists to make individual races easier to manage and to leave room for per-race lore. Unlike the arcane/divine school files, race files are **not** part of the data-driven pipeline - they're plain hand-authored Markdown, edited directly.
- `core/combat/` - `combat.md` (turn structure, action economy, attack/damage rolls, critical hits, and Mythic Actions - the extra action economy for boss-tier/legendary creatures, including Mythical Resistance for shrugging off ongoing effects) and `maneuvers.md` (dashing, shoving, grappling, dodging, etc.).
- `core/magic/` - spellcasting rules split into `magic_overview.md` (universal rules shared by both paths), `minor_magic.md` (small, instinctive Mana effects - minimum 1 Mana, scaling in scale/duration from there - that fall outside a normal spell), and `spell_crafting.md`, with two parallel school trees:
  - `magic/arcane/` - arcane schools (aeromancy, geomancy, hydromancy, pyromancy, shadowmancy) plus `arcane_overview.md`. The 5 school files are **generated** (see below); `arcane_overview.md` is hand-authored.
  - `magic/divine/` - divine schools (benediction, cultivation, invocation, necration, subjugation) plus `divine_overview.md`. Same split: the 5 school files are generated, `divine_overview.md` is hand-authored.
- `core/feats/` - feat categories (general, martial, arcane, hybrid, prestige, skill), indexed by `feats_overview.md`. The 6 category files are **generated**; `feats_overview.md` is hand-authored.
- `core/equipment/` - `armor.md` and `weapons.md` are **generated**; `supplies.md` and `alchemy.md` (crafting oils/bombs/salves from ingredients) are hand-authored.
- `core/exploration/` - travel and downtime layer: `traveling.md` (hex travel, mounts), `downtime.md` (revelry/celebrations), `leadership.md` (followers and cohorts), `dungeon_turns.md` (the 10-minute Turn as the clock for dungeon/site exploration - light tracking and wandering encounters share one party-wide tally). These feed the core loop directly, not optional flavor: Traveling is how the party reaches a dungeon and burns rations, Leadership is how a growing operation delves deeper without resurfacing, and Downtime is where rewards from delving cycle back into progression.
- `core/setting/` - the world of Ressam itself, hand-authored prose, not part of the data-driven pipeline: `setting_overview.md` (welcome blurb, currency, language, and an index of this chapter - the `races_overview.md`/`feats_overview.md` pattern), `calendar.md` (the seasonal year and its festivals), `history.md` (Age of Mythos, the Sundering, Age of Revival, and the Third Age's wars up to 402 RC), `geography.md` (the Mourning Star and twin moons, the continents, and the Mortal Plane/Fields of Nocturne), `nations.md` (the great powers, the nation relations table, and global/local factions), `religions.md` (the faiths practiced across Ressam), and `heroes.md` (legendary figures). Cross-references a lot with `core/character/races/` - each race's own page covers that race's culture and mythology in more depth than fits here.
- `DESIGN_GUIDE.md` - the yardstick for evaluating any mechanic, item, spell, or rule change: five Design Pillars (Realism as "feel, don't simulate," Elegance, Scarcity, Niches, and Distinction between Martials and Casters), followed by Spell Design Guidelines covering damage caps (`Xd8 x Mana`, Pyromancy alone at `Xd10 x Mana`), the no-save-or-suck rule for Overcome spells, summoning being near-exclusive to Necromancy, per-school distinctiveness, Faith always being required for divine magic, healing defaulting to Temporary HP, spells not bypassing mechanics for free, and Gish/spellblade playstyles being a Feat-driven compromise rather than something magic grants directly. Read this before proposing or reviewing changes to `core/magic/`, `core/feats/`, `core/equipment/`, or their `data/` sources - it's the reasoning behind *why* those files are shaped the way they are, not just what's in them.
- `ENCOUNTER_GUIDE.md` - GM-facing reference for calibrating enemy stat blocks against player power, levels 1-12. Codifies "one defensive identity per enemy" (bosses get one reactive option plus one passive trait on top), the four monster archetypes (Brute/Tank/Skirmisher/Caster, each with a primary defence, reaction, weakness, and signature trait), stat-range tables per tier (HP/AR/Evasion/to-hit/damage), and the attack-vs-Evasion scaling math that makes hit rate a flat 67% at every level by design. Grounded in the same formulas as `scripts/formulas.py` and the `core_rules.md`/`combat.md` mechanics they mirror. Consult this (and the `dnd-to-ressam` skill, for porting existing stat blocks) when building or balancing bestiary/encounter content.
- `CREDITS.md` - attribution and licensing note.
- `TODO.md` - planning doc for in-progress, cross-session rework items, when one exists. Only exists while something is actively open; gets deleted once that item is fully resolved and merged, so its absence means "nothing outstanding" rather than an oversight. Check it (and recent git log/diff) before touching feats, combat, or equipment mechanics - a change already in flight may be scoped and reasoned here.
- `assets/` - currently empty (reserved for future images/handouts).
- `data/`, `scripts/`, `templates/` - the data-driven content pipeline; see below.
- `README.md` - short public-facing entry point (project blurb + link to the GitHub Wiki); not the place for pipeline/authoring detail, that lives here.
- `.github/workflows/sync-wiki.yml` - CI that publishes `core/` to the GitHub Wiki; see "Publishing: GitHub Wiki" below.

## Data-driven content pipeline

Weapons, armor, spells, feats, and the character progression table live as structured YAML in `data/` and are rendered into the corresponding `core/**/*.md` files by `scripts/build.py`. This exists because those domains are hundreds of repetitive, data-shaped entries (stat blocks, spell headers, feat prerequisites, per-level advancement rows) that are painful to keep consistent by hand and awkward for tooling/AI to query reliably out of freeform prose.

**Do not hand-edit the generated files** - `core/equipment/weapons.md`, `core/equipment/armor.md`, all 5 files under `core/magic/arcane/` except `arcane_overview.md`, all 5 files under `core/magic/divine/` except `divine_overview.md`, all 6 files under `core/feats/` except `feats_overview.md`, and `core/character/progression_&_rewards.md`. Edits there are silently overwritten the next time someone runs the build. Edit the YAML in `data/` instead (or the `.j2` template directly for the static prose baked into `progression_&_rewards.md.j2`).

- **Data** (source of truth): `data/weapons/*.yaml`, `data/armor/*.yaml`, `data/spells/arcane/*.yaml`, `data/spells/divine/*.yaml`, `data/feats/*.yaml` (six normal category files), `data/feats/prestige.yaml` (the prestige feats, a different shape - no `Benefit`, adds `Ritual`/`Mechanical Changes`), and `data/progression.yaml` (the per-level XP/skill/attribute/feat advancement table).
- **Schema**: `scripts/models.py` (weapons/armor), `scripts/spell_models.py` (spells), `scripts/feat_models.py` (feats), `scripts/progression_models.py` (progression table, with validators that reject non-contiguous levels, XP that doesn't match `xp_per_level * level^2`, or skill/attribute caps that regress across levels) - Pydantic models. Free-text body fields (`effect`, `benefit`, `special`, `mechanical_changes`, feat `extra` sub-fields) store Markdown close to verbatim as YAML block scalars rather than being decomposed further - the internal structure of spell effects and feat benefits (Overcome/Resist pairs, choice-menus, nested sub-effects) varies too much entry-to-entry to model generically, so only the reliably-flat header metadata (name, mana cost, casting time/range/duration, prerequisites) is real structured data.
- **Templates**: `templates/weapons.md.j2`, `templates/armor.md.j2`, `templates/magic/arcane/*.md.j2`, `templates/magic/divine/*.md.j2`, `templates/feats/*.md.j2`, `templates/character/progression_&_rewards.md.j2` - Jinja2. Only the tabular/repeated-entry regions are template-driven; surrounding rules prose (the Weapon Properties glossary, Firearm Rules, school-level passive traits like Shadowmancy's Inaudible, Invocation's Severing ritual, the Prestige Feats Requirements/Restrictions preamble, the Rewards & Treasure and Itemization sections of `progression_&_rewards.md`, etc.) is static text baked into the template. **Exception**: `templates/character/character_sheet.html` is a hand-authored, self-contained interactive player character sheet (plain HTML/CSS/JS, no build step) - it is not Jinja2, not consumed by `scripts/build.py`, and not data-driven from `data/*.yaml` (a player fills in race/career/feats/spells/equipment by hand); it only lives under `templates/` because that directory name also reads correctly for "character sheet template." Open it directly in a browser to use it.
- **Build script**: `scripts/build.py`, run from the repo root as `python scripts/build.py`:
  - no flags: regenerates and writes all `core/**/*.md` output files.
  - `--check`: exits nonzero if any generated output would differ from what's on disk (no write) - use to verify the checked-in `.md` files match `data/`.
  - `--diff`: prints a unified diff of what would change (no write).
  - Workflow: edit YAML in `data/`, run `python scripts/build.py`, commit both the YAML change and the regenerated `.md` diff together.
- **Shared YAML loaders**: `scripts/loaders.py` holds the leaf functions that read `data/` YAML into Pydantic models (`load_yaml`, `load_data`, `load_spell_data`, etc.) - both `scripts/build.py` (generation) and `scripts/balance_report.py` (read-only analysis, see below) import from it, so the analysis tool never depends on the module whose job is to write into `core/`.
- **Python environment**: `.venv/` at the repo root has `PyYAML`, `Jinja2`, and `pydantic` installed (`scripts/requirements.txt`) - run scripts with `.venv/bin/python scripts/build.py` (or activate the venv first).
- **One-time migration scripts**: `scripts/_extract_spells.py` and `scripts/_extract_feats.py` parsed the original hand-authored Markdown into the current YAML during migration. They are not part of the ongoing pipeline (nothing imports or runs them automatically) and are kept only as a reference for how the extraction was done.

## Balance analysis tooling

A separate, read-only layer on top of the same `data/` YAML computes derived-stat progression curves (HP, Evasion, Dodge, AR, Mana, Slots, initiative, and each archetype's *primary* offense bonus - weapon Attack Bonus for martial builds, Spell Modifier for casters - plus Average Ward, a general defensive benchmark) across levels 1-12, for spotting power spikes/dead zones as the game gets rebalanced. It never writes into `core/` - only `scripts/build.py` generates rulebook content.

- **`scripts/formulas.py`**: pure functions mirroring the rulebook's derived-stat formulas (HP, Evasion, Attack Roll bonus, Spell Modifier, Ward DC, Initiative, Slots, etc.). Each function's docstring cites the exact `core_rules.md`/`combat.md`/`magic_overview.md` line it mirrors - when a rule changes in prose, grep for that citation to find what to update here (and vice versa). Deliberately **not** wired into markdown generation: those three files are ~90% narrative prose with formulas scattered throughout, unlike the clean tabular data weapons/armor/progression generate from, so templating them wholesale would trade a small consistency win for a lot of editing friction on every future prose change.
- **`data/builds/*.yaml`** + **`scripts/build_models.py`**: hand-authored archetype characters (Frontline Fighter, Battle Mage, Skirmisher) used as "typical build" stand-ins for the analysis. Each stores starting attributes/skills/equipment plus per-level deltas (not 12 fully-repeated snapshots) - `Archetype.snapshot(level)` folds them into a `CharacterSnapshot`. `check_archetype_budget()` is a soft cross-check (warns, doesn't raise) against `data/progression.yaml`'s point budgets. Battle Mage stands in for both Arcane and Divine casters, since Spell Modifier/attack/overcome math is identical between the two traditions (`magic_overview.md`) - only preparation mechanics (scribing vs. prayer) differ, and those don't affect these stats. To add a fourth archetype, copy the shape of an existing `data/builds/*.yaml` file; `check_archetype_budget()` will warn (not fail the build) if its point spend exceeds `data/progression.yaml`'s budget at any level.
- **`scripts/balance_report.py`**: the CLI. `python scripts/balance_report.py [--format csv|json] [--out PATH] [--levels 1-12] [--archetype KEY ...]`, default CSV to stdout. Emits long/tidy rows (`archetype, level, stat, value`). Refuses any `--out` path under `core/`. Two stats worth knowing about: `primary_offense_bonus` resolves per-archetype (Attack Bonus for martial builds, Spell Modifier for casters) so it's directly comparable across archetypes without special-casing; `average_ward` is `formulas.ward_dc()` averaged across all 8 attributes via the `SKILL_CATEGORIES` map (mirrors the Skill Categories table, `core_rules.md:63-82`) - a general "how hard to affect with magic" benchmark, not any single spell's actual target DC (a real Spell Overcome targets one specific attribute's Ward). No charting library is bundled by design - visualize the CSV/JSON output ad hoc rather than adding a plotting dependency to the repo (see "Visualizing balance_report.py output" below).

### Visualizing balance_report.py output

There's no dashboard checked into the repo - charts are built fresh each time as a self-contained HTML page (a Claude Artifact), using the `dataviz` skill for design guidance. Workflow, in order:

1. `python scripts/balance_report.py --format csv --out <scratchpad>/balance_report.csv` (or `--archetype`/`--levels` to narrow it down) - never point `--out` at anything under `core/`, it's refused anyway.
2. Reshape the long-format CSV into a small embeddable JSON blob keyed `{archetype: {stat: [12 values]}}` - trivial size (a few KB), so it's inlined directly as a JS literal in the page rather than fetched (Artifacts run under a strict CSP that blocks external requests entirely - no CDN chart libraries, no `fetch`).
3. Invoke the `dataviz` skill before writing any chart code - it drives the categorical palette (fixed hue order, not cycled), mark specs (2px lines, 8px end-dots), and the accessibility pass (legend for 2+ series, a hover crosshair+tooltip listing every series at that level, and a plain data table as a non-hover fallback).
4. Two archetypes/stats sharing a panel need distinct visual identity beyond color when they represent different *metrics* (e.g. an archetype's own offense line vs. another's defense line in the same panel) - solid vs. dashed stroke is the pattern used so far, with a small style-key legend explaining what each dash pattern means (separate from the archetype color legend).
5. This dev environment has no Node.js, so `dataviz`'s `scripts/validate_palette.js` can't be run directly to re-verify a categorical subset - fall back to the palette already validated in the skill's `references/palette.md` (its documented slot order, e.g. slots 1-3 for a 3-series chart) rather than picking new hex values by eye.
6. Default to writing the HTML into the session scratchpad and publishing it as an Artifact, not into this repo - it's a point-in-time report, not checked-in rulebook content. Ask first if the user wants a CSV snapshot committed somewhere instead.

## Publishing: GitHub Wiki

The rulebook is published as a GitHub Wiki, auto-synced from `core/` on every push to `main` via `.github/workflows/sync-wiki.yml`: the workflow clones the wiki's own git repo (`<repo>.wiki.git`), mirrors `core/`'s contents into it 1:1 (folder structure preserved, `core/` prefix stripped), and pushes only if something changed. **Never hand-edit wiki pages that correspond to a `core/**/*.md` file** - edits made directly in the wiki UI are silently overwritten on the next sync, the same rule as the generated files above. Edit `core/` (or the YAML in `data/` for generated files) and let CI publish it.

Two pages exist **only** in the wiki repo, not in this one, and *are* meant to be hand-edited directly in the wiki UI: `Home.md` (landing page) and `_Sidebar.md` (persistent nav, shown on every page). The sync workflow explicitly excludes both from its mirror/delete step so it never overwrites them. Both list every chapter grouped by section, in the same order as "Repository structure" above. When a chapter is added, renamed, or removed under `core/`, update both pages to match.

**GitHub Wiki quirk to know before editing either page:** wiki page URLs/links are flat and basename-only, regardless of which subfolder a file is mirrored into - e.g. `core/magic/arcane/pyromancy.md` is mirrored to `magic/arcane/pyromancy.md` in the wiki repo but renders at `/wiki/pyromancy`, not `/wiki/magic/arcane/pyromancy`. Multi-level nested paths in Markdown links (e.g. `[Pyromancy](magic/arcane/pyromancy)`) 404. Always link to wiki pages by bare filename with no folder prefix and no `.md` extension, e.g. `[Pyromancy](pyromancy)`. This only works because every `core/**/*.md` basename is unique repo-wide - if that ever stops being true, this linking scheme needs to be revisited.

## Content conventions

- Em Dashes (—) and En Dashes (–) are not used, instead use only Hyphens (-).
- Files are exported from Google Docs, so literal Markdown-special characters are backslash-escaped in the source (e.g. `1d12 \+ Skill Ranks \+ Attribute`, `Ressam is not Dungeons \& Dragons`, `HP \= (END × 3\) \+ 10`). Preserve this escaping style when editing existing text so re-exports/diffs stay clean. For the generated files (weapons/armor/spells/feats), this escaping lives in the YAML data (and is checked in single-quoted or block-scalar form, since double-quoted YAML strings can't hold a bare `\+`/`\-`/`\=`) - preserve it there when editing data instead.
- Formulas and derived stats are given as bolded inline math (e.g. `**Maximum HP = (END × 3) + 10**`) rather than code blocks. Follow this pattern for new formulas.
- Section breaks use a horizontal rule (`---`) between major topics within a chapter.
- Tables are used heavily for skill lists, DC tiers, and comparative mechanics (e.g. Arcane vs. Divine casting) - match existing table shapes when adding related content instead of introducing prose where a table already exists for that topic.
- Core terminology is reused verbatim across chapters (e.g. `Major Action`, `Ward`, `Evasion`, `AR`, `Exhaustion`, `Mana Cost`) - when editing one chapter, check whether the same term/mechanic is defined or referenced elsewhere (`core_rules.md`, `combat.md`, `magic_overview.md`) so definitions stay consistent.
- The game's tone/scope per `core_rules.md`'s foreword: Ressam is a crunchy, lethal alternative to other TTRPGs, set in a world balanced around an early-1500s baseline - matchlock firearms sit alongside plate armor, magic is feared and regulated. Combat is the heart of the game; everything outside of it is meant to be played loosely, "rulings not rules."
